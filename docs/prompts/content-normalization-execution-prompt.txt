# Execute the Content Normalization + Technology Badges plan

Orchestrate execution of a finished implementation plan on the `redesign` branch of
`~/web/portfolio`. You run the orchestration yourself — inline, in this session —
delegating file work to parallel subagents. You commit; subagents do not.

## Read first
- Plan: `docs/superpowers/plans/2026-05-18-content-normalization.md` — 11 tasks. Code
  tasks (2, 3, 9, 10) contain the **complete file content / exact edits**. Content
  tasks (4–8) carry the **Normalization Rules** (a transformation spec + two worked
  examples) instead of verbatim output — 40 files cannot be inlined.
- Spec: `docs/superpowers/specs/2026-05-18-content-normalization-design.md`.
- Delegation rules: `docs/prompts/subagents-directive.md` — apply to every subagent
  prompt (terse, no preamble, long output → `docs/agents-logs/`).

## Ground rules
- **Do NOT invoke the `subagent-driven-development` or `executing-plans` skills.**
  Orchestrate manually, wave by wave, as below.
- **No browser-based review or verification.** Verification is static only:
  `pnpm astro check` (0 errors), `pnpm build` (succeeds), `pnpm test` (32 pass),
  `grep`.
- Code tasks (2, 3, 9, 10) are transcription — the subagent writes the exact content
  shown in the plan, no redesign. Content tasks (4–8) are transformation — the
  subagent reads each file and applies the Normalization Rules; it does not invent,
  translate, or rewrite case-study prose.
- **Subagents edit files only** (Read/Write/Edit). They do NOT run git, `astro check`,
  or builds. The orchestrator runs all checks, dispatches reviewers, and commits —
  sequentially — after each wave.
- Use each task's model tag: Tasks 2, 3, 9, 10 → `haiku`; Tasks 4–8 → `sonnet`.
- Branch stays `redesign`. Don't push, don't merge.

## Ordering hazard — schema goes LAST among code
Task 9 makes `technologies` a **required** schema field. If it runs before the 40
content files carry that field, `pnpm build` fails. Task 9 is therefore in Wave 2,
strictly after the Wave 1 content tasks. Until then Zod silently strips the unknown
key, so Wave 1's checkpoint is green. No two Wave 1 tasks share a file — full parallel
is safe (Task 3 owns `en.ts`/`es.ts`; nothing else touches them).

## After each wave (orchestrator)
1. `pnpm astro check` — must be 0 errors. After Wave 1 and Wave 2 also run `pnpm build`.
   - Wave 1 checkpoint is green even though `technologies` is not yet in the schema —
     Zod strips it. If `astro check` errors here, a content file has malformed
     frontmatter or body; fix before committing.
2. Dispatch `feature-dev:code-reviewer` agents on the wave's `git diff` — parallelize,
   ~one reviewer per 2–3 tasks. Content reviewers check the Normalization Rules were
   applied (4 sections, `###`/`####` levels, em-dash bullets, no `<b>`, no `>` lists,
   `technologies` extracted faithfully). Code review only, no browser. Per the
   directive: long output → `docs/agents-logs/`, else ≤5 bullets.
3. Fix high-confidence reviewer findings.
4. Commit each task's files **sequentially**, using the plan's commit messages.
5. Tick that wave's checkboxes in the plan file.
